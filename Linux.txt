DAY 11: Linux Architecture & Memory Spaces
- Concept: Separation of Powers - User Space vs Kernel Space.
- Internals: Hardware rings (Ring 0 vs Ring 3), CPU execution modes.
Why dosent one bad app crash your entire computer??
The OS creates two distinct protected worlds 
1 - one for your apps
2 - one for your Kernal

There are two spaces in linux 
1 - User space
Area where application run
has limited access to hardware
user mode is unpriviledged

2 - kernal space
area where OS core executes
Full access to hardware
kernal mode is priviledged

Purpose of having these two spaces
Protection - Prevents user programs from crashing the OS
Security - stops apps from directly accessing memory devices
stability - kernal runs isolated, independent of user apps

system call 
formal request from app to kernal for a service which cannot performed by itself
user space communicates with kernal space via system calls

user space --> system call --> kernal mode --> hardware access --> user mode

Protection rings in operating system
Protection rings are hiearchial security mechanism in modern operating system and cpus that defines different privilege levels for executing code
They prevents user application from directly accessing critical system resources ensuring security stabality and process isolation

levels
ring 0 (kernal mode)
Has the highest privileges and full control of hardware and memory
Executes os kernal functions devices drivers and system services
can run priviledged instructions lile I/O operatios and memory managment

Ring 1 (priviledged mode)
used for some os services like device managment
can access hardware but some restrictions are there compared to ring 0

Ring 2 (semi priviledged mode)
Handles system utilities and middleware that requires controlled hardware access
commonly used for networkig proceess and virtual machine monitors

Ring 3 (user mode)
Has least privileges running user applications
can not directly access hardware or priviledged instructions
needs system calls to os services

The wait() 
system call allows the parent process to pause execution until one of its child processes finishes. 
When the child terminates, the kernel notifies the parent and provides information such as the child’s exit status. 
After the parent collects this information through wait(), the kernel can completely remove the child process from the process table.

Why Zombie processes exist and why they cannot be killed using 'kill'.
You cannot kill something that has already exited. 
The only thing left is its record in the kernel’s process table.
You need to kill parent so zombie process exists
