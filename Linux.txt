MiB Mem (Physical Memory): This represents your actual, physical RAM (Random Access Memory) hardware installed on the server.
MiB Swap (Virtual Memory): This is a designated space on your Hard Drive or SSD (either a dedicated swap partition or a swap file) that the Linux kernel uses as an extension of your RAM.

The Scenario: Imagine a sudden traffic spike hits your application, or a developer writes a piece of code that causes a temporary memory leak.

Without Swap: The moment your RAM hits 100%, the Linux kernel has no choice but to trigger the OOM (Out Of Memory) Killer. It will immediately terminate your primary process (like your Docker container, Java app, or Database) to keep the operating system alive. Your application goes down completely.

With Swap: Swap acts as an overflow cushion. The system slows down because it's using the disk, but it stays alive. It gives your autoscaling policies time to kick in, or gives you (the SRE) time to log in, investigate, and fix the issue before a total outage occurs.

A graceful kill (SIGTERM) politely requests a process to save its data and shut down cleanly, while a forceful kill (SIGKILL) immediately terminates the process through the OS kernel, risking data corruption.

This is also what many systems do automatically. For example, when stopping a container, the runtime first sends SIGTERM and waits for the application to exit gracefully. If it doesn’t exit within the configured timeout, it sends SIGKILL to force termination.

SIGTERM is a graceful termination signal that allows a process to perform cleanup before exiting. SIGKILL is a forceful termination signal sent by the kernel that immediately stops a process without allowing any cleanup or signal handling.

Always send SIGTERM first because it allows the application to shut down gracefully and release resources. If the process does not terminate or is unresponsive, use SIGKILL to force immediate termination.

Virtual File System (VFS)
Each file system (ext4, XFS, FAT32, NTFS, etc.) has its own internal way of storing and managing files on disk. Without the Virtual File System (VFS), applications would need separate code to work with each file system. The VFS provides a common interface, so applications simply call open(), read(), write(), or close(). The VFS then forwards these requests to the correct file system driver.

Why do we need VFS?

Linux supports many file systems such as:

* ext4
* XFS
* FAT32
* NTFS
* NFS

Each file system stores files differently.
If there were no VFS, every application would need to know how each file system works.
For example, a Python program would need different code for:
* ext4
* XFS
* FAT32
* NTFS
This would make programming very difficult.

⸻

What is VFS?
Virtual File System (VFS) is a layer inside the Linux kernel that sits between applications and the actual file systems.
Its job is to hide the differences between file systems.
Applications always communicate with the VFS instead of communicating directly with ext4, XFS, FAT32, etc.

The Python program never checks:

* Is this FAT32?

Instead it simply asks the VFS:

“Please open this file.”
The VFS checks which file system owns that file and forwards the request to the correct file system.

Difference between File System and VFS
File System
* Actually stores files on the disk.
* Examples: ext4, XFS, FAT32, NTFS.
* Knows how to read and write data for its own disk format.

Virtual File System (VFS)
* Does not store any files.
* It is a layer inside the Linux kernel.
* Provides one common interface for all file systems.
* Decides which file system should handle a file request.

An inode stores metadata about a file and pointers to the file’s data blocks. For small files, the inode uses direct pointers. For larger files, it uses single, double, and triple indirect pointers, where each additional level points to blocks containing more block addresses. This allows Linux to efficiently support both small and very large files without increasing the inode size.
An inode has a limited number of direct pointers. When a file grows beyond what those pointers can reference, Linux uses indirect pointer blocks. A single indirect pointer references one block of pointers, a double indirect pointer references a block of pointer blocks, and a triple indirect pointer adds one more level of pointer blocks. This allows Linux to efficiently support very large files without making every inode unnecessarily large.

Suppose I have a log file of 10 GB that is continuously being written to by a Java application. The disk becomes full, so I delete the log file using rm. This removes only the directory entry (filename). However, the application still has the file open through its file descriptor, so the inode and its data blocks remain allocated. Since the inode still exists, the 10 GB of disk space is not freed, and df -h still shows the disk as full. Once the application closes the file or is restarted, the file descriptor is released. As there is no directory entry and no open file descriptor pointing to the inode, Linux deletes the inode and its data blocks, and the disk space is finally reclaimed.

What is a Memory Page?
A memory page is a fixed-size block of memory used by the operating system to manage RAM and virtual memory.

The operating system does not move entire tasks or processes to virtual memory. Instead, it moves memory pages (typically 4 KB each) that are less frequently used. A process can have some pages in RAM and other pages in swap at the same time.

The freed space in RAM is then used to load a memory page that is currently needed by the CPU or an active process. This process is called page replacement.

Page Fault

A page fault occurs when a process tries to access a memory page that is not currently present in physical memory (RAM).

When the CPU executes an instruction, it accesses memory through the Memory Management Unit (MMU). The MMU checks whether the required memory page is present in RAM.
* If the page is present in RAM: Execution continues normally.
* If the page is not present in RAM: A page fault occurs.

The operating system then:
1. Pauses the process.
2. Finds the required page on disk (either in the executable file or in swap space).
3. If RAM is full, selects a page to remove (page replacement).
4. Loads the required page into RAM.
5. Updates the page table.
6. Resumes the process.

Minor Page Fault
A minor page fault occurs when the required memory page is already present in RAM, but the process’s page table does not yet have a valid mapping to it.
Since the page is already in RAM:
* ❌ No disk access is required.
* ✅ The operating system simply updates the page table.
* ✅ Execution continues almost immediately.
It is very fast.

Major Page Fault
A major page fault occurs when the required memory page is not present in RAM.
The operating system must:
1. Pause the process.
2. Read the required page from the SSD/HDD (or swap).
3. Load it into RAM.
4. Update the page table.
5. Resume the process.
Since disk access is required, it is much slower.
